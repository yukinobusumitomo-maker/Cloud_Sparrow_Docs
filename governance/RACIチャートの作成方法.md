# RACIチャートを作る流れ
※本資料では、インフラエンジニアのチームを例に出している

##  🔵 RACI の縦軸（業務領域）＝ “目的 × 手順 × 責任の分岐点” の3つで切るのが最適解
 ※RACIチャートの縦軸は業務領域、横軸は担当者
 
⚠RACI の失敗パターンは
- 業務を細かくしすぎる
- 逆に粗すぎて意味がない
のどちらか。
その中間を作るための黄金ルールがこれ。

### 1.「目的」で大分類を作る（業務の目的別）

#### 例：インフラ・IT運用チームの場合
  - 障害対応
  - 問い合わせ対応
  - 構成変更
  - 定型作業
  - 運用改善
  - プロジェクト作業（任意）

####  ⭐POINT：この“目的ごとの箱”をまず作ると、混乱しない。

### 2. 次に「手順の変わり目」で中分類を作る

#### 例1: 障害対応なら、
  - 切り分け
  - 一時回避
  - 恒久対策
  - 影響範囲調査
  - 事後報告書（顧客向け / 社内向け）

#### 例2: 構成変更なら、
  - 要件確認
  - 設計
  - 承認
  - 変更作業
  - 動作確認
  - ドキュメント更新

####  ⭐POINT：ここはタスクの“流れ”で切る。

### 3.「責任が変わる境界」で最終行（業務行）を作る

これは RACI を作る時に一番重要。
#### 例：
  - 切り分け（誰が判断する？誰が作業する？）
  - 影響範囲調査（〇〇さんが△△の領域に強い → C になりやすい）
  - 設計（得意領域な人 → R / A になりやすい、業務に精通しリスクに敏感な人 → C になりやすい）
  - 承認（作業の影響度によるが基本的に上長が A に抜擢される）
  - 定型作業（手順通りの作業 → 新人でも R があり得る）

##  🔵 結果、RACIの縦軸はこんな構成になる（粒度のちょうどいい例）
◇障害対応  
 ├─ 切り分け  
 ├─ 一時回避策の実施  
 ├─ 恒久対応案の検討  
 ├─ 影響範囲調査  
 └─ 事後報告書（顧客）  

◇構成変更  
 ├─ 要件確認  
 ├─ 設計  
 ├─ 変更申請（承認）  
 ├─ 作業実施  
 └─ 動作確認  

◇定型作業  
 ├─ 月次レポート  
 ├─ リソースメンテナンス  
 └─ 資料更新  

◇運用改善  
 ├─ 課題の棚卸し  
 ├─ 改善案の設計  
 └─ 標準化（手順書 / 自動化）


## 🔵 本題：R / A / C / I の判断基準をどう作るか？

### ◆ R（Responsible）＝ 手と頭を動かす実働担当
判断基準：
  - 実際の作業をする人
  - OJT レベルで担当していい人
  - 手順があれば動ける人
  - 業務の中心に位置する人

### **◆ A（Accountable／最終責任者）だけは毎行に“1人” (最も重要なポイント)**

これは絶対。

担当者が誰であれ。

例：

  「最終的にOKを出すのは承認者（例：チームリーダーや管理者、上長）」という構造なら A は必ず承認者。

ここだけ守れば RACI は崩れない。

### ◆ C（Consulted）＝ “助言者”であり、責任の肩代わりはしない

判断基準：
  - R が判断に迷った時に相談する専門家
  - 経験値が高い人
  - 業務の質に関わる人
  - 決定権は持たない

### ◆ I（Informed）＝ 結果だけ受け取る人

判断基準：
  - 承認者（課長）が A なので、その過程は I
  - 他部署調整担当
  - 多忙で逐一関与すべきでない人
  - 結果だけ知っていれば困らない人
